# CloudNotes Production Incident

CloudNotes is a customer-facing note-taking service. A recent production
deployment is failing to come online. You have been paged to investigate.

This repository is a practical Cloud Engineering assessment. It runs completely
offline using the Python standard library only. No Docker, Terraform, cloud
accounts, external APIs, or paid services are required.

## Assignment Overview

You are the on-call engineer for the CloudNotes service. The application starts
but does not become ready to accept traffic. Your job is to investigate the
deployment, diagnose the failures, correct the configuration, and complete a
production readiness review of the architecture.

The assessment covers:

- Linux investigation
- DNS troubleshooting
- TLS understanding
- Network diagnostics
- Distributed systems reasoning
- Virtualization concepts
- Cloud responsibility understanding
- Architecture planning

## Repository Layout

```text
cloudnotes-production-incident/
├── app.py                      # Service bootstrap and simulated startup
├── requirements.txt
├── config/
│   ├── app.conf                # Application settings
│   ├── dns.conf                # Service host / DNS settings
│   └── tls.conf                # TLS settings
├── certificates/
│   ├── server.crt              # TLS certificate
│   └── server.key              # TLS private key
├── logs/
│   └── application.log         # Runtime logs (generated by app.py)
├── architecture/
│   └── cloudnotes-production-architecture.md
├── reports/
│   └── production-readiness.md # Review to be completed by you
└── README.md
```

## Startup Instructions

Run the service:

```bash
python3 app.py
```

The application will attempt to start, resolve its service host, and validate
its TLS configuration. Diagnostic output is printed to the terminal and written
to `logs/application.log`.

## Investigation Instructions

1. Run the application and observe that startup does not complete.
2. Inspect `logs/application.log` to see the reported failures.
3. Examine the configuration under `config/` and the files in `certificates/`.
4. Use standard Linux and network diagnostic tools to investigate each failure.
5. Correct the configuration so the service reports that it is ready to accept
   traffic.
6. Review `architecture/cloudnotes-production-architecture.md` and complete
   every section of `reports/production-readiness.md` with your findings and
   recommendations.

## Deliverables

- A corrected, healthy startup (the application reports it is ready to accept
  traffic).
- A completed `reports/production-readiness.md`.
